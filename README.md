## データ生成パイプライン（Step1〜Step4）

### Step1: XMLファイルの取得
法令APIから、司法試験主要7科目の対象法令を固定リスト方式で取得する。

- 対象科目
  - 憲法
  - 民法
  - 商法
  - 民事訴訟法
  - 行政法
  - 刑法
  - 刑事訴訟法
- 行政法は集合概念のため、行政手続法・行政不服審査法・行政事件訴訟法・国家賠償法などを個別に検索
- 候補絞り込み
  - repeal_status が廃止相当なら除外（取得できる場合）
  - amendment_type が 1 を優先（取得できる場合）
  - 複数候補は公布日最新を採用（全部改正対策）
  - amendment_type/repeal_status がAPI v1で取得できない場合は、現行法一覧前提で最新公布日フォールバックを採用し、理由をマニフェストに記録
- 出力
  - 生XML: xml_raw 配下
  - 取得ログ: fetch_manifest_step1.json

### Step2: 生データの正規化
取得XMLからタイピングに適した問題文を項単位で抽出し、正規化・フィルタリングする。

- 抽出粒度
  - 項単位（Paragraph）
- 除外条件
  - Item タグを含む Article はスキップ（号列挙の条を除外）
  - 正規化後に 号 を含む文を除外
- 正規化条件
  - NFKC 正規化
  - カッコ内除去 + 残存カッコ記号除去
  - 文字数 50〜130 のみ採用
  - 重複除外（field/source/text）
- 追加修正（品質改善）
  - Ruby の Rt（ふりがな）を本文抽出から除外（瑕疵かし のような重複混入を防止）
- 出力
  - 正規化データ: normalized_questions_step2.json
  - 集計ログ: normalize_manifest_step2.json

### Step3: かな変換
正規化済みデータを、ひらがなの問題データへ変換する。

- 変換前処理
  - NFKC 正規化
  - 固有読みマップを優先適用
    - 遺言 -> いごん
    - 競売 -> けいばい
    - 瑕疵 -> かし
    - 勾留 -> こうりゅう
- 変換方式
  - pykakasi を使用
  - 固有読みはプレースホルダ方式で先に保護し、かな変換後に復元
- 追加修正
  - text と kana の末尾を句点で統一（。）
- 出力
  - かな変換データ: questions_step3_kana.json
  - 集計ログ: kana_manifest_step3.json

### Step4: JSONへ変換
最終配布用JSONを確定形式で生成する。

- 最終形式
  - questiondata 配列
  - 各要素は text, kana, field, source を保持
- 最終整形
  - 不正レコード除外
  - 重複除外
  - 件数・科目別件数をマニフェスト化
- 出力
  - 最終データ: questions.json
  - 集計ログ: finalize_manifest_step4.json
